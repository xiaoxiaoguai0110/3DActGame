## 今日讨论要点

### 1. 输入系统的架构设计

**讨论问题**：新 Input System 的输入处理应该怎么组织？

**结论**：采用 **InputReader 单例 + 混合模式**

| 输入类型 | 处理方式 | 原因 |
|----------|----------|------|
| **移动（WASD）** | 每帧读属性 `MoveInput` | 持续输入，值变化不频繁但使用频繁 |
| **攻击/跳跃/格挡** | 事件驱动 `OnAttack` | 瞬时输入，一次性触发即可 |

**工程感悟**：
- 移动和跳跃是两种不同类型的输入，适合不同的处理方式
- 不要一刀切全部用事件或全部用轮询，要看场景

### 2. 单例模式（MonoBehaviour 版）

**核心模板**：

```csharp
public static ClassName Instance { get; private set; }

void Awake()
{
    if (Instance != null)
    {
        Destroy(gameObject);  // 已有老大，新来的滚蛋
        return;
    }
    Instance = this;
    DontDestroyOnLoad(gameObject);
}
```

**关键理解**：
- `if (Instance != null)` 判断是为了防止场景切换时重复创建
- 销毁的是**新来的**，不是旧的，保证已有引用不会断
- `DontDestroyOnLoad` 让这个对象跨场景存活

**面试考点**：
- 单例的优缺点
- 为什么不能直接 `Instance = this` 覆盖
- 多线程环境下单例的线程安全问题

### 3. 组件概念

**核心理解**：
- Unity 中挂载到 GameObject 上的每一个脚本/功能块都是**组件**
- `MonoBehaviour` 继承类 = 组件
- 组件化的好处：一个 GameObject 可以有多个组件各司其职

### 4. 事件驱动 vs 每帧轮询的性能误区

**核心结论**：两者性能差距可以忽略不计

- 事件：**只在值变化时触发**，按住 W 不放只触发一次
- 轮询：`ReadValue<Vector2>()` 本质是读缓存，开销极小
- 选择依据应该是**代码清晰度**和**控制权**，而不是性能
